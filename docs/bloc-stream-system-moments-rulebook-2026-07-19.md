# Bloc Stream System Moments Rulebook - 2026-07-19

This is the product/backend blueprint for real Bloc Stream system moments.

System moments are not sample content. They must be generated by canonical
backend writes, stored as real stream messages, and protected by idempotency
keys so retries, polling, and merge/replay work cannot duplicate them.

## Core Contract

- System moments are rows in the same stream as chat messages.
- `message_type = 'system'`.
- `author_id = null`.
- The actor, target member, month, linked workout, linked settlement, or linked
  event lives in structured `payload`, not in display-only text.
- UI labels/body can be recomputed from payload if copy changes later.
- Moments are Bloc-scoped and visible only to current Bloc members.
- Historical moments remain visible after a member leaves only if the app later
  chooses to support former-member history. For the current product, stream
  read access should follow active Bloc membership.

## Required Storage Shape

Recommended fields for `bloc_messages`:

```text
id uuid primary key
bloc_id uuid not null
message_type text not null -- text | system | event
author_id uuid null
body text null
system_kind text null
payload jsonb not null default '{}'
created_at timestamptz not null default now()
idempotency_key text null
```

Recommended uniqueness:

```text
unique (bloc_id, idempotency_key)
where idempotency_key is not null
```

System moments must be inserted with an idempotency key. The backend should use
`insert ... on conflict do nothing` semantics.

## Ordering Rules

- Moments sort by `created_at` ascending in the stream.
- If a system moment is caused by a user action, use the action commit time.
- Month-close moments should use the actual close/finalization time, not the
  last day at midnight.
- Do not retroactively generate old moments during normal reads. Backfills must
  be explicit admin/migration jobs.

## Moment Kinds

### `member_joined`

Trigger:

- A user successfully joins a Bloc through invite/join flow.

Idempotency key:

```text
member_joined:{bloc_id}:{member_user_id}
```

Payload:

```json
{
  "memberUserId": "...",
  "memberDisplayName": "...",
  "joinedAt": "...",
  "joinedMonthKey": "2026-6"
}
```

Copy intent:

- Label: `NEW MEMBER`
- Body: `{name} joined the Bloc.`

Do not emit:

- During read-side membership repair.
- During display-name repair.
- During historical import/backfill unless explicitly requested.

### `member_left`

Trigger:

- A member leaves a Bloc voluntarily.

Idempotency key:

```text
member_left:{bloc_id}:{member_user_id}:{left_at_iso_or_membership_revision}
```

Payload:

```json
{
  "memberUserId": "...",
  "memberDisplayName": "...",
  "leftAt": "..."
}
```

Copy intent:

- Label: `LEFT BLOC`
- Body: `{name} left the Bloc.`

Do not emit:

- For hidden legacy cleanup.
- For duplicate leave retries.

### `member_removed`

Trigger:

- An admin removes/kicks a member.

Idempotency key:

```text
member_removed:{bloc_id}:{target_user_id}:{removed_at_iso_or_membership_revision}
```

Payload:

```json
{
  "actorUserId": "...",
  "targetUserId": "...",
  "targetDisplayName": "...",
  "removedAt": "..."
}
```

Copy intent:

- Label: `REMOVED`
- Body: `{name} was removed from the Bloc.`

Do not expose the admin name in copy unless the product explicitly wants that
social dynamic.

### `target_hit`

Trigger:

- A workout log causes a member's current-month count to cross from below their
  current member target to greater than or equal to target.

Idempotency key:

```text
target_hit:{bloc_id}:{month_key}:{member_user_id}
```

Payload:

```json
{
  "memberUserId": "...",
  "memberDisplayName": "...",
  "monthKey": "2026-6",
  "count": 12,
  "target": 12,
  "logId": "..."
}
```

Copy intent:

- Label: `TARGET HIT`
- Body: `{name} hit target.`
- Sub: use a contextual line only when true, e.g. `First to target this month.`

Do not emit:

- If the member was already at/above target before the log.
- For deleted/restored/refetched logs.
- For historical logs outside the active month unless the product deliberately
  supports retroactive moments.

### `first_to_target`

This can be represented either as a separate `system_kind` or as
`payload.firstToTarget = true` on `target_hit`.

Trigger:

- Same as `target_hit`, and no other active member in the Bloc had already hit
  target for that month.

Idempotency key:

```text
first_to_target:{bloc_id}:{month_key}
```

Recommendation:

- Prefer storing it on the `target_hit` payload so the stream has one moment,
  not two back-to-back moments for the same workout.

### `perfect_bloc_month`

Trigger:

- The month is finalized and every active, non-excused member hit their member
  target for that closed month.

Idempotency key:

```text
perfect_bloc_month:{bloc_id}:{month_key}
```

Payload:

```json
{
  "monthKey": "2026-5",
  "memberCount": 5,
  "targetSummary": {
    "Aadhil": 12
  }
}
```

Copy intent:

- Label: `PERFECT BLOC MONTH`
- Body: `Everyone hit target.`

Do not emit:

- For active in-progress month.
- If anyone active and non-excused missed their target.

### `month_closed`

Trigger:

- A month is finalized/closed and results are available.

Idempotency key:

```text
month_closed:{bloc_id}:{month_key}
```

Payload:

```json
{
  "monthKey": "2026-5",
  "winnerUserIds": ["..."],
  "loserUserIds": ["..."],
  "settlementCount": 3,
  "action": "season_results"
}
```

Copy intent:

- Label: `MONTH CLOSED`
- Body: `{Month} ended - summary ready.`
- Sub: `{n} to pay` when settlements exist.

UI behavior:

- Tapping opens the relevant month results/settlement screen.

### `monthly_awards`

Trigger:

- Immediately after month close, once closed-month stats are calculated.

Idempotency key:

```text
monthly_awards:{bloc_id}:{month_key}
```

Payload:

```json
{
  "monthKey": "2026-5",
  "awards": [
    { "title": "Bloc Champ", "userId": "...", "displayName": "..." },
    { "title": "Most consistent", "userId": "...", "displayName": "..." },
    { "title": "Biggest turnaround", "userId": "...", "displayName": "..." },
    { "title": "Furthest behind", "userId": "...", "displayName": "..." }
  ]
}
```

Copy intent:

- Label: `AWARDS · {MONTH}`
- Render with the existing awards card UI.

Tie handling:

- `Bloc Champ`: include all tied winners or use the same winner convention as
  the Month/Settlement screen. Do not invent a different ranking.
- `Most consistent` and `Biggest turnaround`: only emit when the metric is
  defined cleanly. If not available, omit that award from payload.
- `Furthest behind`: omit if everyone hit target.

### `inactivity_warning`

Trigger:

- An active, non-excused member has no workout logs for 7 full days in the
  current month and has not already received the same warning recently.

Idempotency key:

```text
inactivity_warning:{bloc_id}:{month_key}:{member_user_id}:{window_start_date}
```

Payload:

```json
{
  "memberUserId": "...",
  "memberDisplayName": "...",
  "monthKey": "2026-6",
  "daysInactive": 7,
  "windowStartDate": "2026-07-12"
}
```

Copy intent:

- Label: `INACTIVITY`
- Body: `{name} - no workout in 7 days.`

Rate limit:

- Maximum one inactivity warning per member per 7-day window.
- Do not emit after the member is mathematically cooked; the cooked moment is
  stronger and should replace this.

### `cooked`

Trigger:

- A current-month member becomes mathematically unable to hit target with the
  remaining days in the month, assuming max one counted workout per day if that
  is the product rule.

Idempotency key:

```text
cooked:{bloc_id}:{month_key}:{member_user_id}
```

Payload:

```json
{
  "memberUserId": "...",
  "memberDisplayName": "...",
  "monthKey": "2026-6",
  "count": 2,
  "target": 12,
  "daysLeft": 3
}
```

Copy intent:

- Label: `COOKED`
- Body: `{name} can't reach target this month.`
- Sub: `Fine locked at month end.`

Do not emit:

- For excused/sit-out members.
- More than once per member per month.

Retraction/backfill correction:

- If a member was marked cooked because they had not logged workouts yet, and
  later backfills enough workouts in the same active month to become no longer
  cooked, delete the prior cooked system moment.
- This is a correction to stale stream state, not a comeback story.
- The retraction key is the same idempotency key:

```text
cooked:{bloc_id}:{month_key}:{member_user_id}
```

- Only apply this automatic deletion while the month is still active/open.
  Historical month repair should be an admin repair path, not normal product
  behavior.

### `comeback`

Trigger:

- A member moves from `behind` to `on-track` in the active current month after
  logging.
- `behind` must use the app's current member pace status rule. As of this note,
  runtime member pace is:
  - `on-track`: `count >= expected`
  - `at-risk`: `count >= expected - 2`
  - `behind`: anything below `expected - 2`, unless cooked
- Product shorthand may describe this as being roughly `-4 behind pace`, but
  the implementation must use the actual shared pace-status helper/threshold in
  the app.
- Do not emit comeback for `at-risk -> on-track`; that is too mild.
- Do not emit comeback for `cooked -> on-track`; that is handled by deleting
  the stale cooked moment when backfilled logs make the member no longer cooked.

Idempotency key:

```text
comeback:{bloc_id}:{month_key}:{member_user_id}:behind:on-track
```

Payload:

```json
{
  "memberUserId": "...",
  "memberDisplayName": "...",
  "monthKey": "2026-6",
  "fromStatus": "behind",
  "toStatus": "on-track",
  "logId": "..."
}
```

Copy intent:

- Label: `COMEBACK`
- Body: `{name}: Behind -> On Track.`

Rate limit:

- Maximum one comeback per member per month, unless the product later wants
  repeat comeback stories.

### `final_stretch`

Trigger:

- There are 3 days left in the current month and at least one active,
  non-excused member is short of target.

Idempotency key:

```text
final_stretch:{bloc_id}:{month_key}
```

Payload:

```json
{
  "monthKey": "2026-6",
  "daysLeft": 3,
  "shortMemberUserIds": ["..."]
}
```

Copy intent:

- Label: `FINAL STRETCH`
- Body: `3 days left. {n} members still short.`

Do not emit:

- If every active, non-excused member has already hit target.

### `event_created`

Event cards are user-created stream messages, not system moments.

Use:

- `message_type = 'event'`
- `author_id = creator`
- `payload.activity`
- `payload.when`
- `payload.location`
- `payload.rsvp`

Do not create a separate `system` moment for event creation unless analytics
later show it is useful.

## Non-Goals For V1

- No fake seed messages in preview/live.
- No AI-generated moments.
- No automatic tone-policing/copy generation.
- No retroactive moment generation during normal GET/read.
- No Realtime requirement for V1. Polling/revision refresh is acceptable first.
- No public table access without a deliberate RLS/Data API audit.

## Backend Implementation Plan

1. Add real stream tables:
   - `bloc_messages`
   - `bloc_message_reactions`
   - `bloc_message_reads`
2. Add a backend helper:
   - `insertSystemMoment({ blocId, kind, payload, idempotencyKey, createdAt })`
3. Call the helper from successful canonical write paths only:
   - `join-group`
   - `leave-bloc`
   - `kick-member`
   - `add-log`
   - `multi-log`
   - `update-settings`
   - `sitout-request`
   - `sitout-review`
   - settlement claim/confirm/dispute if product-approved
   - month-close/finalization path when that becomes explicit
4. Bump the canonical revision clock after stream writes that should refresh
   clients.
5. Keep stream reads id-keyed:
   - backend stores user IDs
   - frontend resolves display names from Bloc membership/profile data
6. Replace `src/lib/blocStream.js` in-memory calls with API calls while keeping
   the existing UI contract.

## QA Matrix

- New real Bloc opens with no fake messages.
- Sending a text message persists after refresh.
- Reacting to a message persists after refresh and does not reorder chips
  incorrectly.
- Joining a Bloc creates exactly one `member_joined` moment.
- Logging the target-reaching workout creates exactly one `target_hit` moment.
- Deleting/refetching the same workout does not duplicate or remove unrelated
  moments.
- Month close creates exactly one `month_closed`, one optional
  `monthly_awards`, and one optional `perfect_bloc_month`.
- Sit-out approval creates exactly one `sit_out_approved`.
- Leaving/kicking does not reveal private Stream data to non-members.
